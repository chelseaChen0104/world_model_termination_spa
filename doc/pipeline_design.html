This pipeline goes from a raw Sudoku environment all the way to a trained termination predictor that beats SPA baselines.

Stages at a glance

  • Stage 1 — Data generation (two tracks: random-play CPU + LLM-policy GPU)
  • Stage 2 — Supervised finetuning (SFT) — produces world-model + termination tags
  • Stage 3 — Reinforcement learning (RL) — lifts BP recall via asymmetric reward
  • Stage 4 — Evaluation — vs SPA baselines + heuristic

Why two data tracks?

Track A is a fast CPU-only intermediate that lets us debug the full pipeline before spending GPU hours on Track B. Track B is the headline data — it matches SPA's self-play recipe and produces in-distribution training samples.

What we contribute on top of SPA

The termination tags <solvable>, <breaking_point>, <steps_left> are layered onto SPA's existing <observation> + <prediction> training shape. Same self-play pipeline, extended target.

There are three distinct format flows in this pipeline. They must not be conflated — each has different strictness requirements. See doc/SPEC.md §7.5 for the locked decisions.

A · LLM data-gen output LOCKED

Only <answer> is required to be parseable. Everything else is forgiving.

System prompt (data gen, minimal):

You are playing Sudoku on a 9x9 grid. ...
Output your move EXACTLY in this format:
<answer>place N at row R col C</answer>
Output ONLY the <answer>...</answer> tag.

Why minimal: the base Qwen2.5-1.5B has never seen our format. Asking for 6 tags drops parse rate to ~30–50% and slows decode 10–20×. Asking for only <answer> gives >90% parse rate.

Track quality: parse_failure_rate per trajectory in the per-attempt log. Threshold: >30% = data is effectively random-play.
B · SFT target row LOCKED

All designed tags, content from env ground truth, not LLM hallucinations. Mirrors SPA §2.2.

<think>
<observation>{step.state}</observation>       ← env
<prediction>{step.next_state}</prediction>     ← env
<terminate_prob>0.10</terminate_prob>
<steps_left>{bucket}</steps_left>               ← oracle
<solvable>true</solvable>                       ← oracle
<breaking_point>false</breaking_point>          ← oracle
</think>
<answer>{step.action_name}</answer>

Built by SFTFormatter.format_step() for every training row, regardless of what the LLM produced. SPA's recipe: replace beliefs with ground truth.

Anti-goal: changing this tag set or order without re-running all baselines. The XML is the API between SFT, RL, and eval.
C · Multi-turn prior turns OPEN

When the prompt at step K includes prior assistant turns from steps 0…K−1, what content do we use?

// Two valid policies:

[1] LLM-raw (current with full prompt)
    prior = step.llm_raw_response  (may be malformed)
    → matches inference distribution
    → realistic noise in priors

[2] Template-clean (current with minimal prompt)
    prior = SFTFormatter.format_step(step)
    → consistent prior + target shape
    → cleaner training signal

With the minimal data-gen prompt, llm_raw_response is set to None on the trajectory step. SFTFormatter then falls back to the template (policy [2]). This is the current behavior.

Decision is open. If we ever switch back to the full data-gen prompt, the policy flips to [1] and we'd want to track parse_failure_rate as the data-quality signal.

How they connect

During data gen, the LLM produces (A). The resulting action is parsed and fed to the env. The env returns ground-truth state + labels. The training row is then constructed as (B), with priors as (C). The LLM's actual XML output (other than the action) is essentially discarded for training purposes — its job is to produce diverse, in-distribution actions, not to write the training format.

This separation is what makes "minimal data-gen prompt + full SFT target" a valid optimization: we get cheap, fast, high-quality data gen AND the full training signal at the SFT step.

Per doc/SPEC.md §3, the headline result must compare against SPA's baseline set on Sudoku, with both Pass@k metrics (for SPA comparability) and termination metrics (our contribution).

Metrics

  • Pass@1, Pass@8 — fraction of puzzles solved in the highest-probability rollout / any of K rollouts. From SPA Table 2. TODO
